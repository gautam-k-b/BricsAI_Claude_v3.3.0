## 🚀 v3.3.0 — Two-Phase Batch Mapper, One-by-One Review & Prompt Caching
*Released: April 2, 2026*

This release is a performance and UX milestone. The layer mapping pipeline has been re-architected from a sequential per-layer loop into a two-phase batched LLM pipeline, reducing mapping time for 100+ layer drawings from ~5 minutes to ~36 seconds. The human-in-the-loop review workflow is now incremental — proposals appear one at a time — and every proposal now includes a plain-English LLM-generated reason. Prompt caching is also now properly exploited by moving all dynamic content to the bottom of both SurveyorAgent and ExecutorAgent system prompts.

### 🛡️ Booth Layer Protection — Proofing Sequence Reordered

**Architectural Fix:** `NET:LOCK_BOOTH_LAYERS` has been moved from the **end** (step E) to the **beginning** (step 0) of the proofing sequence. This ensures booth layers are locked **before** `NET:PREPARE_GEOMETRY` runs, preventing any risk of them being exploded during the geometry recursive elimination pass.

**Updated Sequence:**
```
0. NET:LOCK_BOOTH_LAYERS           (protect booth layers FIRST)
A. NET:PREPARE_GEOMETRY            (safe; booth layers are locked)
B. NET:APPLY_LAYER_MAPPINGS        (migrate vendor objects to A2Z layers)
C. (c:a2zcolor)                    (enforce standard colors)
D. (command "-PURGE" ...) + NET:RENAME_DELETED_LAYERS
```

**ComClient Guard:** If a proofing batch is detected but `LOCK_BOOTH_LAYERS` is missing from the LLM response, the engine now pre-executes it at the start of the batch (via a new "Proofing pre-guard") rather than attempting to append it at the end. This guarantees booth layers are protected throughout the entire sequence.

### ⚡ Two-Phase Batched Mapper (Major Performance Improvement)

**Before:** The Mapper polled COM geometry and called the LLM once per unknown layer. For 100 unknown layers this meant 100 sequential LLM round-trips (~5 min total).

**After:** `MapperAgent` now runs two phases:

1. **Phase 1 — `ClassifyByNameAsync`**: Sends all unknown layer names in a **single LLM call**. Layers with recognisable names (e.g. `A-WALL`, `G-COLS`, `BOOTH-NUM`) are classified immediately as confident mappings. Layers with ambiguous names (e.g. `LAYER1`, `MISC`, numeric codes) are flagged as `UNCERTAIN` for Phase 2.

2. **Phase 2 — `BatchDeduceByGeometryAsync`**: Only `UNCERTAIN` layers are polled via `NET:POLL_LAYER_SEMANTICS`. All their footprints are then sent to the LLM in **one batched call**. The LLM returns a `{ "mappings": [...] }` JSON object for all uncertain layers in one shot.

**Outcome:** Most drawings need only 2 LLM calls total for the entire mapping pipeline, regardless of layer count. Typical time reduced from ~5 min to ~36 sec.

**JSON Object mode fix:** Both batch methods explicitly use a `{ "mappings": [...] }` root object schema (not a bare array) to comply with the Azure OpenAI SDK's `CreateJsonObjectFormat()` enforcement. Parsers use `TryGetProperty("mappings", ...)` to extract the array.

### 🔍 One-by-One Mapping Review

The `MappingReviewAgent` now presents mapping proposals **one at a time** instead of all at once.

- Each proposal shows the source layer, proposed target, and a short LLM-generated reason sentence
- The drafter responds with a natural-language reply; `ClassifySingleMappingResponseAsync` classifies it as `ACCEPT`, `SKIP`, or `ABORT`
- Accepted mappings are queued; skipped ones are bypassed without being persisted
- `ABORT` halts the review immediately, preserving all mappings accepted so far
- The performance summary (token count, elapsed time) is displayed **before** the first proposal, not after

### 💬 LLM-Generated Evidence Reasons

Every mapping proposal now includes a plain-English reason sentence generated by the LLM itself (not constructed from raw entity counts). The reason is extracted from the `reason` field in the LLM's JSON response and stored in `_mappingReasons` for display in the review chat bubble.

Example: *"312 lines and polylines typical of wall and partition geometry."*

### 🔒 Quick Action Buttons Disabled During Review

A new `IsQuickActionsEnabled` property (`!IsBusy && !_isInOneByOneMappingReview`) controls the three Quick Action sidebar buttons (**Run Full AI Proofing**, **Clean Geometry**, **Generate Summary**). They are disabled for the entire duration of a mapping review session, preventing accidental re-triggers. The Send button remains independently controlled by `IsNotBusy`.

### 💾 OpenAI Prompt Caching Optimisation

Dynamic content injected from `KnowledgeService.GetLearnings()` has been moved to the **bottom** of both `SurveyorAgent` and `ExecutorAgent` system prompts. This maximises the stable token prefix eligible for OpenAI's automatic server-side prompt cache (50% discount on input tokens for prefixes ≥ 1024 tokens). Previously, injecting dynamic content mid-prompt invalidated the cache on every run.

## 🚀 v3.2.0 - Proofing Reliability, Layer Safety & Booth Inspection

This release closes a class of silent failures where the AI generated plausible-but-wrong sequences and where locked layers blocked read-only inspection commands. Every fix has a corresponding C# hard-stop so the behaviour is enforced at the engine level, independent of whatever the LLM generates.

### 🛡️ Proofing Sequence Enforcement (ComClient Guards)

**Problem:** The LLM occasionally generated entirely wrong proofing batches — omitting `NET:RENAME_DELETED_LAYERS` and `NET:LOCK_BOOTH_LAYERS`, adding hallucinated commands like `(c:a2zLayers)` and `(c:a2zLayouts)`, or prepending stray `NET:LEARN_LAYER_MAPPING` calls before `NET:PREPARE_GEOMETRY`.

**Fix — three independent guards in `ComClient.ExecuteActionAsync`:**

1. **Proofing completeness guard** — Before iterating a tool-calls batch, the engine pre-scans for `APPLY_LAYER_MAPPINGS`. If found (indicating a proofing run), it also checks for `RENAME_DELETED_LAYERS` and `LOCK_BOOTH_LAYERS`. Any missing mandatory step is auto-appended **after** the main loop with a `PROOFING GUARD` log entry, so the drawing always ends up in the correct final state.

2. **LEARN_LAYER_MAPPING block in proofing batches** — Any `NET:LEARN_LAYER_MAPPING` command that appears inside a proofing batch is silently skipped and logged as `GUARD: Skipped unsolicited LEARN_LAYER_MAPPING inside proofing batch`. This prevents the LLM from permanently writing mapping rules it inferred from Surveyor hints rather than from an explicit user instruction.

3. **Existing DELETE-after-RENAME guard** — Unchanged: `NET:DELETE_LAYERS_BY_PREFIX` is blocked in the same batch as `NET:RENAME_DELETED_LAYERS`.

### 🤖 Executor Agent Rule Hardening

- **Rule 7 — Anti-hallucination proofing rule:** `NET:LEARN_LAYER_MAPPING` explicitly added to the list of commands forbidden inside a proofing sequence. New note: "If the Surveyor context mentions that certain `Expo_` layers should be 'categorised' or 'mapped', that is informational only — do NOT convert it into LEARN_LAYER_MAPPING calls."
- **Rule 7 — Forbidden extra steps:** `(c:a2zLayers)`, `(c:a2zLayouts)`, raw duplicate `_.EXPLODE` calls, and any command not in steps A–E are explicitly listed as forbidden.
- **Rule 10 — LEARN_LAYER_MAPPING standalone-only:** Rewritten to state the command is only valid as a direct response to a user instruction, never during a proofing sequence and never sourced from Surveyor hints.
- **Rule 8c — COUNT_EMPTY_BOOTHS:** New rule mapping all "count / investigate / how many unnumbered booths" phrasings to exactly `NET:COUNT_EMPTY_BOOTHS`. Explicitly forbids the hallucinated variants `NET:COUNT_UNMAPPED_BOOTH_BOXES` and `NET:COUNT_UNMATCHED_BOOTH_BOXES`.

### 🔍 Surveyor Agent — Exact Layer Name Matching

**Problem:** The Surveyor classified layers like `Expo_BoothOutline MAX` as standard because they started with `Expo_BoothOutline`. This caused the Mapper to skip them and the LLM to treat them as known targets.

**Fix:** New **EXACT MATCH RULE** added to the Surveyor system prompt: a layer is only standard if its name matches the table entry character-for-character, including case. `Expo_BoothOutline MAX` is now correctly tagged `[UNKNOWN]` and routed through the Mapper.

### 📊 New Command: `NET:COUNT_EMPTY_BOOTHS`

A read-only inspection command that determines how many booth outlines in `Expo_BoothOutline` do not contain a corresponding booth number from `Expo_BoothNumber`.

**Implementation** (`GeometryToolsPlugin`, V15 + V19):
- Uses COM `SelectionSet.Select(mode=5, filterCode=8, filterValue="Expo_BoothOutline")` to collect all outline polylines by layer name — no area heuristic, no entity type guessing
- Uses a parallel selection set filtered to `TEXT,MTEXT` on `Expo_BoothNumber` to collect all number insertion points
- Runs a ray-casting point-in-polygon test for each text point against each outline polygon
- Returns: `Booth outline count: X. Booth number count: Y. Booths WITHOUT a booth number: Z. Booths WITH a booth number: W.`
- Count-only — never highlights or moves objects, so it is safe to run on locked layers

**Known limitation:** BricsCAD COM `SelectionSet.Select` returns 0 objects when the target layer is locked. Because `LOCK_BOOTH_LAYERS` runs at the end of every proofing sequence, `NET:COUNT_EMPTY_BOOTHS` will return all-zero counts unless layers are unlocked first. This is tracked for the next release.

### 🔒 LockBoothLayers — Exact Name Matching Confirmed

Audited `LockBoothLayers` in both `LayerToolsPlugin` (V15 + V19). The implementation uses `doc.Layers.Item("<ExactName>").Lock = true` with four specific exact strings, so it cannot accidentally lock a layer like `Expo_BoothOutline MAX`. No code change was needed — this is noted here for clarity.

### 🧹 Clean Geometry — Elapsed Time Display

The **Clean Geometry** quick action now shows elapsed time in the completion message (e.g. `Clean Geometry completed in 4.2s`). Token count is shown as `0` for this fast-path since it does not invoke the LLM.

### 📝 Documentation

- `README.md` fully rewritten: accurate architecture diagram, complete `NET:` command reference table for both plugin families, proofing sequence A–E with C# guard descriptions, A2Z layer table with exact-match rule, `agent_knowledge.txt` format, and layer lifecycle diagram.

## 🚀 v3.1.0 - The Architecture & Stabilization Update

This patch focuses on bulletproofing legacy V15 LISP execution desyncs and introduces the formal BricsAI Executive Architecture documentation for stakeholders.

### 🛡️ LISP Synchronization & `ssget` Defensive Guards
- **The COM Desync Patch**: Resolved a critical V15 bug where `_.CHPROP` layer assignments against manually locked or empty vendor layers caused BricsCAD to silently drop the command early. This resulted in subsequent arguments (like `_LA` and `Expo_Markings`) bleeding into the command line as unrecognized rogue input.
- **Strict `ssget` Wrapping**: All `_.CHPROP` executions inside the `LayerToolsPlugin` are now strictly encapsulated within native `(if (setq ss (ssget ...)))` LISP guards. - **Absolute Pre-Unlocking**: Decoupled the instant C# hardware unlock loop (`ForceUnlockAllLayersSynchronously()`) from the Semantic Polling Agent and hardcoded it to the absolute start of `ExecuteQuickAction` in `MainViewModel.cs`. The instant a Drafter clicks "Run AI Proofing", BricsCAD violently strips **all** physical locks globally before the Agents even look at the file.

### 📊 Executive Architecture & Token Economics
- **Investor Deck Generation**: Shipped a comprehensive Markdown and Mermaid presentation specifically designed for non-technical stakeholders. It details the "Mixture of Experts" pipeline, the separation of the C# Engine from the LLM, and the exact Agent interactions.
- **ROI Token Metrics Analysis**: Computed the precise OpenAI GPT-4o Token expenditure vs. physical Drafter labor values. The documentation validates how BricsAI processes a drawing for `< $0.02` per execution, reducing a traditional 2-hour ($70) proofing workflow down to 0.5 - 1.0 hours of high-level QA, yielding a mathematically proven 50% - 75% reduction in task expenditure.

## 🚀 v3.0.0 - Semantic Mapping & Human-in-the-Loop Interception

This massive update elevates BricsAI from a static Automation Script into a dynamically learning, conversing CAD Architect. It introduces active semantic heuristics, mathematical CAD protections, and an interactive UI mapping orchestrator.

### 🧠 Semantic Mapper & Interactive Dashboard
- **Semantic Polling Telemetry**: The Engine now actively intercepts any unknown vendor layers *before* proofing starts, invoking a dedicated mapper tool (`NET:POLL_LAYER_SEMANTICS`) that physically counts blocks and texts natively across the BricsCAD COM interface to geometrically deduce the vendor's intent.
- **Human-in-the-Loop Review**: If new mapping structures are deduced, the application gracefully suspends its execution thread and alerts the Drafter in the chat. The user can naturally type "Proceed" to auto-run the macros, or converse with the AI ("Change X to Y") to manipulate the pending JSON payload before it hits the drawing.
- **Token Performance Tracking**: The Chat UI now natively logs and aggregates the specific OpenAI Token consumption across the entire Surveyor and Mapping pipeline, printing performance metrics directly to the screen upon completion or abort.

### 🛡️ Preemptive Hardware Unlocking & Precision Heuristics
- **Asynchronous COM Desync Fix**: Resolved a critical bug where rapid C# processing evaluated target logic faster than BricsCAD could asynchronously unlock layers, destroying protected data. The system now utilizes a strict sequential Pre-Unlock execution phase entirely prior to LISP evaluation.
- **Synchronous Surveyor Unblocking**: Because Drafters often deliver the AI manually-locked CAD files, the `ComClient` now executes `ForceUnlockAllLayersSynchronously()` natively instantly before Semantic Polling engages, completely preventing geometric observation failures.
- **Mathematical Booth Protection Heuristic**: Beyond JSON dictionaries, the C# Plugins now feature an indestructible string-matching fallback heuristic. If a layer physically contains the words "booth outline" or "booth number" (while strictly avoiding "max" variants), compiling constraints force a physical `Layer.Lock = true` clamp natively over COM before the drawing explodes.

### 🧰 BricsCAD Robustness Upgrades
- **Deleted Entity Carry-Over Resiliency**: Discovered that when BricsCAD renames a globally locked layer to `Deleted_...`, it duplicates the Lock state, trapping the entities forever. The system now injects a `layer.Lock = false` physical reset natively inside the deprecation loop to guarantee clean final delivery.
- **Anti-Crash Polling Architecture**: Wrapped the primary LLM COM Executor (`SendMessageAsync`) inside a global exception dispatcher, ensuring that HTTP API timeouts, JSON Parse errors from stray BricsCAD quotes, or COM disconnects gracefully dump standard `⚠` errors to the Chat interface instead of violently crashing the WPF pipeline.
- **Intelligent Prefix Resolution**: Removed fragile prefix exclusion algorithms. The C# Engine now deterministically builds the "Unknown String Array" entirely offline using native Layer Collections compared strictly against standard dictionaries, providing mathematically perfect payloads to the Semantic auto-mapper.

## 🌟 v2.0.0 - The Multi-Agent Orchestrator Update

This massive update shifts BricsAI from a simple LLM command generator to a fully autonomous, multi-agent reasoning engine capable of executing complex 6-step proofing workflows with near-perfect reliability.

### 🤖 Multi-Agent Pipeline
- **Surveyor Agent**: Automatically analyzes drawing state, extracting layer counts and bounding box heuristics before touching geometry.
- **Executor Agent**: Synthesizes the Surveyor's data to generate a massive array of LISP and Native COM macros.
- **Validator Agent**: Reads the execution logs directly from BricsCAD to verify that all commands theoretically executed as requested.

### 🏗️ Native COM Selections & Migrations
- **LISP ssget Replacement**: Completely bypassed the `ssget` command's selection bleed-over issues.
- **Native Target Instantiation**: The C# Host now explicitly checks for and creates target layers on the fly (e.g., `Expo_BoothOutline`), directly assigning geometric `.Layer` properties mathematically via DXF 8 filtering.
- **Safe Explosion Constraints**: Blocks are iteratively exploded while explicitly safeguarding primitives (Polylines, Arcs, Text) from destruction.

### ⚙️ Configurable Layer Mappings
- **Dynamic Vendor Mapping**: Introduced persistent learned rule storage (`agent_knowledge.txt` via `KnowledgeService`). Users can now map bizarre source CAD layers (e.g., `l1xxxx`) directly to standard A2Z layers (e.g., `Expo_BoothOutline`) and have it persist across runs.
- **Zero-Code Adaptation**: Learned mappings are injected into both the Surveyor and Executor prompts at runtime, forcing absolute mapping prioritization without needing to recompile the AI.

### 🧹 Destructive Cleanup Tooling
- **Safe Deletion (legacy)**: Initially introduced the `(command "-LAYDEL" ...)` tool logic to forcefully annihilate empty vendor layers that resist standard `PURGE`. As of **v2.0.1**, this path has been fully superseded by the license-safe deep deletion pipeline described above (no Express Tools dependency).

### 🛠️ Developer Notes
- **Pluggable Tool Plugins**: Introduced the `IToolPlugin` contract and versioned plugin assemblies (`BricsAI.Plugins.V15Tools`, `BricsAI.Plugins.V19Tools`) loaded at runtime by `PluginManager` based on the detected BricsCAD major version.
- **Multi-Agent COM Host**: Finalized the standalone .NET 9 WPF overlay (`BricsAI.Overlay`) that talks to BricsCAD exclusively over COM, removing any requirement for `NETLOAD` inside the CAD session.
- **Configurable Layer Memory**: Formalized `agent_knowledge.txt` (via `KnowledgeService`) plus the `NET:LEARN_LAYER_MAPPING:<SourceLayer>:<TargetLayer>` command so user edits persist across runs.
